## Инструкция по сборке APK-файла из GitHub-репозитория

Для сборки APK-файла вашего Android-приложения из GitHub-репозитория с использованием командной строки, следуйте этим шагам:

### 1. Установите необходимые инструменты

Убедитесь, что у вас установлены следующие инструменты:

*   **Git**: Для клонирования репозитория.
    *   [Скачать Git](https://git-scm.com/downloads)
*   **Java Development Kit (JDK)**: Android Studio и Gradle требуют JDK.
    *   [Скачать JDK](https://www.oracle.com/java/technologies/downloads/)
*   **Android SDK**: Включает инструменты командной строки, такие как `sdkmanager` и `adb`.
    *   Проще всего установить его, скачав [Android Studio](https://developer.android.com/studio) (SDK устанавливается вместе с ней).
    *   Убедитесь, что переменные среды `ANDROID_HOME` (или `ANDROID_SDK_ROOT`) и `JAVA_HOME` настроены правильно, и пути к каталогам `platform-tools` и `tools` Android SDK добавлены в переменную `PATH` вашей системы.

### 2. Клонируйте репозиторий с GitHub

Откройте терминал или командную строку и выполните следующую команду, заменив `<URL_ВАШЕГО_РЕПОЗИТОРИЯ>` на фактический URL вашего проекта на GitHub:

```bash
git clone https://github.com/Dealtors/Luke.git
```

Например:

```bash
git clone https://github.com/Dealtors/Luke.git
```

### 3. Перейдите в директорию проекта

После клонирования репозитория перейдите в корневую директорию вашего проекта, где находится файл `build.gradle` (для проекта):

```bash
cd Luke
```

### 4. Соберите APK-файл

Для сборки APK-файла используйте оболочку Gradle (`gradlew`).

#### Для отладочной (debug) версии APK:

Эта команда сгенерирует отладочный APK-файл, который не подписан для публикации, но отлично подходит для тестирования.

```bash
./gradlew assembleDebug
```
(В Windows: `gradlew assembleDebug`)

#### Для релизной (release) версии APK:

Для создания релизной версии APK, которая может быть опубликована, вам потребуется настроить подписание приложения. Это обычно включает создание файла ключей (`.jks`) и настройку его в файле `build.gradle` вашего модуля приложения (обычно `app/build.gradle`).

Если подписание настроено, выполните:

```bash
./gradlew assembleRelease
```
(В Windows: `gradlew assembleRelease`)

**Примечание**: Если у вас не настроено подписание для релизной версии, вы получите ошибку или сборка завершится без подписанного APK. Для простого тестирования рекомендуется использовать `assembleDebug`.

### 5. Найдите сгенерированный APK-файл

После успешной сборки APK-файл будет находиться в подкаталоге вашего проекта. Обычно путь к нему выглядит так:

*   Для отладочной версии:
    `app/build/outputs/apk/debug/app-debug.apk`
*   Для релизной версии:
    `app/build/outputs/apk/release/app-release.apk` (или `app-release-unsigned.apk`, если не было подписано)

Вы можете использовать команду `ls` (или `dir` в Windows), чтобы проверить содержимое этих папок.

## Руководство пользователя

Приложение «Luke» предназначено для помощи волонтерам и сотрудникам пунктов сбора.

### 1. Запуск и выбор роли
При первом запуске откроется экран с выбором роли:
- **«Я хочу помочь»** — режим волонтера.
- **«Я представляю пункт сбора»** — режим сотрудника.

### 2. Режим волонтера
1. Выберите **город** из выпадающего списка.
2. Ниже отобразятся все пункты помощи выбранного города.
3. Нажмите на пункт, чтобы увидеть подробную карточку:
   - 📍 Адрес
   - 🕒 График работы
   - 📦 Список потребностей (товары, одежда, медикаменты и т.д.)
   - 📊 Прогресс сбора (сколько собрано / сколько нужно)
4. **Иконка карты** 🗺️ открывает экран с:
   - Метками пунктов сбора
   - Переключением города
   - Просмотром описания пункта при нажатии на маркер
   - Построением маршрута до выбранного пункта
   - Отображением текущей погоды в регионе
5. **Иконка профиля** 👤 содержит:
   - Статистику (сколько раз помогли)
   - Историю действий
   - Редактирование личных данных
6. **Кнопка «Выйти»** → возврат на стартовый экран выбора роли.

### 3. Режим пункта сбора
1. После входа открывается главный экран пункта с перечнем **текущих товаров / потребностей**.
2. **«Добавить товар»** ➕ — создание новой позиции (название, количество, срочность).
3. **«Изменить данные»** ✏️ — редактирование профиля пункта (адрес, контакты, график).
4. В карточке каждого товара доступны действия:
   - ✏️ Изменить количество
   - ⚠️ Изменить срочность (низкая / средняя / высокая)
   - 🗑️ Удалить товар
5. **«Выйти»** → возврат на экран выбора роли.

### 4. Дополнительные особенности
- 📱 **Локальная база данных** (SQLite) — все данные сохраняются даже после перезапуска приложения.
- 🌐 Для работы карты и погоды требуется доступ к **Интернету**.
- ☁️ Погода загружается с внешнего API по защищенному протоколу.

## План монетизации и публикации

Приложение публикуется как **социальный сервис** для помощи волонтерам и пунктам сбора.  
Базовый функционал остается **полностью бесплатным**.

### Возможные способы монетизации
1. **Пожертвования** от пользователей и партнеров (добровольные взносы).
2. **Гранты и поддержка НКО** — финансирование от некоммерческих организаций.
3. **Платные партнерские размещения** для проверенных пунктов сбора (повышенная видимость).
4. **B2B-подписка** для организаций на расширенную аналитику и экспорт данных.

### План публикации
| Этап | Описание |
|------|----------|
| 1 | Подготовка финальной версии APK / AAB |
| 2 | Тестирование на нескольких устройствах (разные версии Android) |
| 3 | Публикация в **Google Play** и **RuStore** |
| 4 | Сбор обратной связи от первых пользователей |
| 5 | Регулярные обновления (исправление ошибок, новые функции) |

## Средства безопасности

Приложение разработано с учетом базовых принципов защиты данных пользователей.

### Реализованные меры
- 🔒 **HTTPS** для всех внешних запросов (API погоды, карты).
- 💾 **Локальная SQLite-база** — данные не отправляются на внешний сервер.
- 🚫 **Минимальные разрешения** — только `INTERNET` для карты и API.
- 🧹 **Хранение данных только на устройстве** — никаких облачных синхронизаций без согласия.
- ✅ **Проверка ввода** на экранах регистрации и редактирования (защита от пустых / некорректных значений).

### Что нужно добавить для production-версии
В учебной версии аутентификация упрощена. Перед промышленным релизом необходимо:
1. **Хэшировать пароли** (например, bcrypt или PBKDF2).
2. Внедрить **нормальную авторизацию** (сессии / токены).
3. Ограничить **доступ к чувствительным данным** (разграничение ролей на серверной стороне).

## Дополнительная информация

### Практическая работа 5. План публикации приложения

Приложение является **социально значимым сервисом** для взаимодействия волонтеров и пунктов сбора помощи.  
Базовый функционал целесообразно оставить бесплатным, чтобы не ограничивать доступ пользователей.  
В качестве модели монетизации рассматриваются:
- добровольные пожертвования,
- грантовая поддержка,
- партнерские интеграции с проверенными организациями,
- расширенная аналитика для корпоративных клиентов.

Публикация планируется через официальные магазины **Google Play** и **RuStore** после завершения тестирования.  
Перед публикацией необходимо проверить:
- корректность интерфейса,
- работу карты и погоды,
- стабильность локальной базы данных,
- обработку внешнего API.

После выпуска приложение будет развиваться за счет исправления ошибок, обновления данных и добавления новых возможностей.
